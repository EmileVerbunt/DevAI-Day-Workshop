What are Agentic Workflows?

15 minutes

GitHub Agentic Workflows (gh-aw) are AI-powered automation that can understand context, make decisions, and take meaningful actions — all from natural-language instructions you write in markdown. Instead of brittle scripts, you describe what you want to happen and let a coding agent figure out how.

Agentic vs. Traditional Automation

Traditional CI automation follows fixed if-then rules. Agentic workflows use coding agents (like Copilot CLI, Claude, or Codex) that go further:

  • Understand context — Read your repository, issues, and pull requests to grasp the current situation.
  • Make decisions — Choose appropriate actions based on context, not just predefined conditions.
  • Adapt behavior — Respond flexibly to scenarios you didn't explicitly program for.
The key idea

A traditional workflow is code that runs. An agentic workflow is natural language that an AI agent runs — compiled into a hardened GitHub Actions workflow so it executes safely and repeatably.

How It Works at a Glance

An agentic workflow is a single markdown file with two parts:

  • Frontmatter — A YAML block at the top that configures when the workflow runs and what it is allowed to do (triggers, permissions, safe-outputs).
  • Markdown body — Your natural-language instructions describing the task.

The gh aw compile command converts this markdown into a hardened .lock.yml GitHub Actions workflow that runs an AI agent in a containerized environment whenever the trigger fires.

What You Can Automate

  • Issue triage — Read new issues, ask clarifying questions, apply labels.
  • Code review — Summarize PRs and flag risky changes.
  • Release management — Draft release notes from merged PRs.
  • Maintenance — Keep docs, changelogs, and dependencies tidy.

Many ready-made examples live in the Agentics collection.

Installing gh-aw & Workflow Anatomy

20 minutes

The gh aw CLI extension lets you create, compile, and run agentic workflows directly from your terminal.

Prerequisites
  • GitHub CLI (gh) installed and authenticated — cli.github.com
  • A repository with GitHub Actions enabled and permission to add workflows
  • A coding agent / model available to your repo (e.g., GitHub Copilot)

Install the Extension

# Install the gh-aw CLI extension
gh extension install github/gh-aw

# Verify the installation
gh aw version

# (Optional) pin a version for team consistency
gh extension install github/gh-aw@v0.1.0

Initialize a Repository

gh aw init prepares the repository for agentic workflows — it configures .gitattributes and scaffolds supporting skill/agent files.

# Prepare the repo (non-interactive defaults)
gh aw init

Anatomy of a Workflow File

Workflows live as markdown files under .github/workflows/. Here is a minimal example:

---
on:
  issues:
    types: [opened]
permissions: read-all
safe-outputs:
  add-comment:
---

# Triage New Issue

Analyze the current issue. If it is unclear or missing key
details (steps to reproduce, expected vs. actual behavior),
post a friendly comment asking for the missing information.

The YAML frontmatter (between the --- markers) controls execution; the markdown body is the agent's task in plain language.

Tip

You can start from a curated example instead of a blank file with gh aw add, which pulls in a workflow from the Agentics collection and generates its compiled YAML.

Mini Challenge (2 min): Read the frontmatter

Look at the minimal example above. In one sentence each, answer: when does this workflow run, what can it read, and what is the single thing it's allowed to write back? Being able to answer these three at a glance is the core safety skill for gh-aw.

Triggers, Permissions & Safe-Outputs

25 minutes

Giving an AI agent access to your repository demands a strong security model. The frontmatter is where you keep agents on a short, well-defined leash.

Triggers — on:

The on: block defines when a workflow runs, using the same event syntax as GitHub Actions.

on:
  issues:
    types: [opened, reopened]
  schedule:
    - cron: "0 9 * * 1"   # every Monday at 09:00 UTC
  workflow_dispatch:        # allow manual runs

Permissions — Read-Only by Default

Agentic workflows default to read-only permissions. The agent can read context but cannot directly write to your repository.

permissions: read-all

Safe-Outputs — Controlled Writes

Because the agent has no direct write access, all write operations flow through safe-outputs: a set of sanitized, validated GitHub operations. The agent proposes an output; gh-aw validates and performs it.

safe-outputs:
  add-comment:          # post a comment
  add-labels:
    allowed: [bug, question, needs-info]
  create-issue:         # open a new issue
  create-pull-request:  # open a PR with changes

This separation means even if the agent is led astray, it can only perform the narrow, pre-approved actions you declared — never arbitrary writes.

Principle of least privilege

Declare only the safe-outputs a workflow actually needs, and constrain them (e.g., an allowed label list). Access can also be gated to team members so untrusted contributors can't trigger privileged agent actions.

Why This Matters

  • No direct write access — Reduces blast radius if an agent misbehaves.
  • Validated operations — Outputs are sanitized before they touch your repo.
  • Auditable — Each action is explicit in the frontmatter and visible in run logs.
Mini Challenge (3 min): Tighten the permissions

Imagine an agent that only needs to label issues. Write the safe-outputs block that grants the least privilege required — just add-labels with an allowed list — and nothing more. Then ask yourself: what's the worst a compromised prompt could do with this configuration?

Step 4 of 5

Compiling & Running

20 minutes

A markdown workflow doesn't run directly — it is compiled into a hardened GitHub Actions workflow first. Think of it like compiling code: you write human-friendly markdown, the compiler produces machine-ready YAML.

Compile to a Lock File

# Compile every markdown workflow in the repo
gh aw compile

For each name.md, gh-aw generates a name.lock.yml — the compiled workflow GitHub Actions actually runs. Compilation:

  • Validates your frontmatter configuration.
  • Applies security hardening (sandboxing, restricted permissions).
  • Embeds the frontmatter and loads the markdown body at runtime.
Commit the lock file

Commit both the .md source and the generated .lock.yml. The lock file is what runs in Actions; keeping it in version control makes runs reviewable and reproducible. Recompile whenever you change the markdown.

Run and Observe

Once committed, the workflow triggers automatically on its configured events. You can also dispatch and inspect runs from the CLI:

# Trigger a run (push pending changes first if needed)
gh aw run --push

# Inspect logs for a workflow's runs
gh aw logs

Each run spins up an AI agent in a containerized environment, which reads the repository context, performs the task described in your markdown, and emits any declared safe-outputs.

The Lifecycle

write .md   ->  gh aw compile  ->  .lock.yml committed
                                       |
                trigger fires (issue, schedule, manual)
                                       |
                  agent runs in container, reads context
                                       |
                       safe-outputs validated & applied
Mini Challenge (2 min): Why compile?

The .md file is what you edit, but the committed .lock.yml is what actually runs. In your own words, why does gh-aw compile to a locked YAML file instead of running the markdown directly? (Hint: think about reproducibility and reviewing exactly what will execute.)

Hands-on: Build an Issue-Triage Agent

30 minutes

Put it all together by building an agentic workflow that triages newly opened issues: it asks for missing details and applies an appropriate label.

The Goal

When an issue is opened, the agent should read it, decide whether it has enough information, comment asking for anything missing, and apply one label from an allowed set.

Step-by-Step

  1. Prepare a test repository with Actions enabled, then run gh aw init.
  2. Create .github/workflows/issue-triage.md with the workflow below.
  3. Compile it with gh aw compile and commit both files.
  4. Open a deliberately vague test issue and watch the agent respond.
---
on:
  issues:
    types: [opened]
permissions: read-all
safe-outputs:
  add-comment:
  add-labels:
    allowed: [bug, enhancement, question, needs-info]
---

# Issue Triage Agent

Read the newly opened issue's title and body.

1. Decide the most fitting label from the allowed set and
   apply exactly one.
2. If the issue lacks reproduction steps, expected behavior,
   or environment details, post one friendly comment listing
   precisely what's missing. If it is complete, do not comment.

Be concise and welcoming in any comment you write.

Verify the Behavior

  • A vague issue receives a polite comment requesting specifics and a needs-info label.
  • A well-formed bug report receives a bug label and no unnecessary comment.
  • The run logs (gh aw logs) show the agent's reasoning and the safe-outputs it emitted.

Now Choose Your Own Agent

You've built the issue-triage agent step by step. Now pick one of the use-cases below and build it yourself using the same pattern: a trigger, read-only permissions, a narrow set of safe-outputs, and a clear natural-language prompt. Each one is a realistic agent teams actually run.

1. Documentation Drift Detection

Goal: Catch docs that fall out of sync with the code they describe.

  • Trigger: pull_request (on changes to source files), or a weekly schedule:.
  • What it does: Compares changed code against related docs/README sections and flags mismatches (renamed functions, changed flags, removed options).
  • Safe-outputs: add-comment on the PR listing the drift, or create-issue for scheduled scans.
  • Stretch: Have it propose the corrected doc snippet inline.

2. Test Quality Analyzer (AAA Pattern)

Goal: Nudge tests toward a clear Arrange–Act–Assert structure.

  • Trigger: pull_request on changes to test files.
  • What it does: Reviews added/modified tests and flags ones with no clear arrange/act/assert separation, multiple assertions testing unrelated things, or missing edge cases.
  • Safe-outputs: add-comment with specific, line-referenced suggestions.
  • Stretch: Apply a needs-test-cleanup label when issues are found.

3. Security Scanner

Goal: Surface likely-exploitable issues early, with low noise.

  • Trigger: pull_request, gated to changed code files.
  • What it does: Looks for hard-coded secrets, unvalidated input reaching sensitive sinks, missing authz checks, and unsafe deserialization — reporting only high-confidence findings.
  • Safe-outputs: add-comment with severity and rationale; optionally add-labels: [security-review].
  • Stretch: Require a maintainer ack before the PR can merge.

4. Duplicate Code Scanner

Goal: Spot copy-paste and near-duplicate logic worth refactoring.

  • Trigger: pull_request, or a weekly schedule: across the repo.
  • What it does: Identifies blocks that are structurally similar to existing code and suggests where a shared helper could live.
  • Safe-outputs: add-comment pointing at the duplicate locations, or create-issue with a refactor proposal.
  • Stretch: Have it draft the extracted helper function in the comment.
Build checklist

For whichever you pick: (1) start from permissions: read-all, (2) allow only the safe-outputs you actually need, (3) keep the prompt focused on one job, (4) compile with gh aw compile, and (5) test against a deliberately flawed example so you can see it fire.

Extend it

Add a weekly schedule: trigger that summarizes stale issues, or a PR-review workflow that flags risky diffs. Browse the Agentics collection for inspiration and copy a workflow with gh aw add.
