What is Spec-Driven Development?

15 minutes

Spec-Driven Development (SDD) flips the usual AI coding workflow. Instead of prompting an agent to "just build it" and hoping the result matches your intent, you first capture what you want and why in a clear specification — then let Copilot turn that spec into a plan, tasks, and code.

Intent vs. Implementation

Traditional prompting mixes intent and implementation together in a single throwaway message. Spec-driven development separates them:

  • Intent (the spec) — The durable description of the problem, the desired behavior, constraints, and acceptance criteria. It is reviewed, versioned, and shared.
  • Implementation (the code) — A consequence of the spec, generated by the agent and regenerated when the spec changes.

When the spec is the source of truth, the code becomes easier to regenerate, review, and reason about — and the AI has a stable target to aim at.

Why Specs Matter for AI Coding

  • Better results — Agents perform dramatically better with explicit requirements than with vague one-liners.
  • Shared understanding — A spec is reviewable by humans before a single line of code is written.
  • Reproducibility — The same spec can drive consistent regeneration across models and sessions.
  • Guardrails — Constraints and acceptance criteria keep the agent from wandering off scope.
Mental model

Think of the spec as the contract and the agent as the contractor. You refine the contract until it is unambiguous; the contractor builds to it.

When to Use SDD

  • Use it for new features, multi-file changes, and anything you'll need to review, hand off, or revisit.
  • Skip it for trivial edits, quick experiments, or single-line fixes where the overhead isn't worth it.

The SDD Loop

  1. Specify — Describe what and why (not how).
  2. Plan — Let Copilot propose a technical approach.
  3. Break down — Decompose the plan into concrete, ordered tasks.
  4. Implement — Execute the tasks, verifying against acceptance criteria.

Over the next steps you'll practice this loop first with plain Copilot, then with GitHub Spec Kit — a toolkit that formalizes each stage with slash commands.

Try it (2 min): Intent vs. implementation

Take a feature you recently built and write a single sentence describing its intent (the what and why) with zero mention of how it was built. If you can't avoid naming a function, framework, or data structure, rewrite it. This is the muscle SDD trains.

Step 2 of 5

Writing Specs with Copilot

20 minutes

Before reaching for dedicated tooling, you can practice spec-driven development with nothing more than a markdown file and GitHub Copilot. The goal: a spec precise enough that Copilot can implement it without guessing.

Anatomy of a Good Spec

A useful feature spec is structured and testable. A simple template:

# Feature: Shopping Cart Discount Codes

## Problem
Customers can't apply promo codes at checkout, so marketing
campaigns can't offer time-limited discounts.

## Goals
- Let a customer enter a discount code in the cart
- Validate the code server-side and apply the discount
- Show the updated total before checkout

## Non-Goals
- Stacking multiple codes
- Per-user usage limits (future work)

## Behavior
- A valid percentage code reduces the subtotal by its percentage
- An expired or unknown code shows an inline error and no discount
- The applied discount persists until the cart is emptied

## Acceptance Criteria
- [ ] Entering `SAVE10` reduces a $100 subtotal to $90
- [ ] Entering `EXPIRED` shows "This code is no longer valid"
- [ ] Removing the code restores the original total
Describe what, not how

Keep specs focused on behavior and outcomes. Leave the implementation details (frameworks, file layout, algorithms) to the planning step so the agent can choose the best fit for your codebase.

Use Plan Mode to Turn a Spec into a Plan

Copilot's Plan Mode reads your spec and the surrounding codebase, then proposes an implementation plan for you to review before any code is written. This is the bridge between intent and implementation.

  1. Open your repository in VS Code with the spec file visible.
  2. Start a Copilot Chat session in Plan Mode and reference the spec: Plan the implementation of #file:cart-discounts.md
  3. Review the proposed steps, push back on anything that doesn't fit, and iterate until the plan matches your intent.

Persist Conventions with Instructions Files

Specs describe a single feature; instructions files describe how all work in the repo should be done. Together they give the agent both the "what" and the "house rules".

Try it (3 min): Sharpen one acceptance criterion

Look at the discount-code spec above. Add one more acceptance criterion that pins down an edge case it currently misses (for example: what happens when a code is applied to an empty cart?). Phrase it as a concrete, checkable assertion — a value in, a value out.

# .github/copilot-instructions.md

- Use TypeScript with strict mode for all new code.
- Co-locate unit tests next to the file under test as `*.test.ts`.
- Validate all external input at the boundary; never trust client data.
- Prefer pure functions; keep side effects in clearly named modules.

You can also scope guidance with path-specific .instructions.md files so different parts of the codebase follow different rules.

Exercise: Write and Plan a Spec

  1. Create specs/feature.md using the template above for a small feature in a sample repo.
  2. Add a .github/copilot-instructions.md with 3–4 conventions for the repo.
  3. Ask Copilot in Plan Mode to plan the implementation from your spec.
  4. Refine the spec until the generated plan is correct — notice how clearer specs produce better plans.

Introducing GitHub Spec Kit

20 minutes

GitHub Spec Kit is an open-source toolkit that turns the spec-driven loop into a repeatable workflow. It adds a set of slash commands to your AI coding agent and a project structure that keeps specs, plans, and tasks organized.

What Spec Kit Gives You

  • A consistent structure — Dedicated locations for your project constitution, specs, plans, and task lists.
  • Slash commands — /speckit.constitution, /speckit.specify, /speckit.plan, /speckit.tasks, and /speckit.implement that drive each stage of the loop.
  • Agent-agnostic — Works with GitHub Copilot and 30+ other coding agents.
Prerequisites
  • Python 3.11+ and uv installed
  • Git and a GitHub Copilot-enabled agent (e.g., VS Code or Copilot CLI)

Install the specify CLI & Initialize a Project

Install the specify CLI with uv, then scaffold a project and choose your agent with --integration. Replace vX.Y.Z with the latest release tag:

# Install the CLI (requires uv)
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@vX.Y.Z

# Scaffold a new project wired up for GitHub Copilot
specify init my-project --integration copilot
cd my-project

Spec Kit adds its /speckit.* commands and a .specify/ + specs/ structure. In IDE agents the commands appear as slash commands; in GitHub Copilot CLI you select the agent with /agents (or address it directly in a prompt).

Define the Project Constitution

The constitution captures the non-negotiable principles your project must always follow — quality bars, architectural rules, testing expectations. The agent consults it during planning and implementation.

/speckit.constitution Create principles for a small web API: all
endpoints must have integration tests, prefer simple
dependency-free solutions, validate every external input,
and document each public function.

Spec Kit writes these principles to a constitution file so every subsequent /speckit.plan and /speckit.implement stays aligned with them.

Try it (5 min): Scaffold and inspect

Run specify init on a throwaway project and open the generated .specify/ and specs/ folders. Then run a /speckit.constitution command with two principles of your own and open the resulting constitution file. What did Spec Kit add that you didn't type?

Constitution vs. instructions

A Spec Kit constitution plays a similar role to a copilot-instructions.md file — project-wide rules — but it is woven into Spec Kit's structured planning commands so the principles are enforced at each stage.

Step 4 of 5

The Spec Kit Workflow

25 minutes

With a constitution in place, Spec Kit drives the rest of the loop through its /speckit.* commands. Each one produces a reviewable artifact you can refine before moving on.

/speckit.specify — Describe What and Why

Start with the feature, focused on intent rather than the tech stack. Spec Kit generates a structured specification file.

/speckit.specify Build a task board where users can create
projects, add tasks to a project, drag tasks between "To Do",
"Doing", and "Done" columns, and see each project's progress.
Focus on the experience, not the implementation.

/speckit.plan — Choose the Technical Approach

Now provide your stack and constraints. Spec Kit produces a technical plan that respects both your spec and your constitution.

/speckit.plan Use a Vite + React frontend with TypeScript, a
lightweight REST API in Node/Express, and in-memory storage
for now. Keep components small and add integration tests for
the API endpoints.

/speckit.tasks — Break the Plan into Steps

Decompose the plan into an ordered, actionable task list. This is where the work becomes concrete and trackable.

/speckit.tasks

Review the generated tasks for correct ordering and dependencies. Reorder or split anything that's too large before implementing.

/speckit.implement — Execute the Tasks

Have the agent work through the task list, building the feature against the spec and constitution.

/speckit.implement
Iterate at the right level

If something is wrong, fix the earliest artifact that's wrong — update the spec, re-plan, regenerate tasks — rather than patching generated code by hand. Keeping the spec as the source of truth is what makes the workflow repeatable.

Optional Quality Commands

Spec Kit adds commands that act like "unit tests for your spec" — use them to catch gaps before you generate code:

  • /speckit.clarify — Surface and resolve underspecified areas (run before /speckit.plan).
  • /speckit.analyze — Cross-artifact consistency & coverage check (run after /speckit.tasks, before /speckit.implement).
  • /speckit.checklist — Generate a custom quality checklist validating requirement completeness, clarity, and consistency.
Try it (5 min): One command in isolation

Before the full hands-on, practice a single stage on its own. In a project that already has a spec, run just /speckit.tasks and read the generated task list. Are the tasks ordered correctly? Is any single task too big to implement safely? Reorder or split one before moving on.

The Full Loop at a Glance

/speckit.constitution  ->  project principles (once per project)
/speckit.specify       ->  what & why  (per feature)
/speckit.clarify       ->  resolve ambiguities (optional)
/speckit.plan          ->  how (stack, architecture, constraints)
/speckit.tasks         ->  ordered, actionable steps
/speckit.analyze       ->  consistency & coverage check (optional)
/speckit.implement     ->  code that satisfies the spec

Hands-on: Build a Feature Spec-First

30 minutes

Put the full loop together. You'll take a small feature from idea to working code entirely spec-first, using Spec Kit.

The Challenge

Build a command-line "habit tracker" that lets a user add daily habits, mark them complete for today, and view a weekly streak summary. No UI framework required — keep it simple.

Step-by-Step

  1. Initialize a fresh project with specify init habit-tracker --integration copilot.
  2. /speckit.constitution — Require unit tests for all logic, no external dependencies beyond the standard library, and clear help output.
  3. /speckit.specify — Describe the habit tracker's behavior and acceptance criteria (add, complete, weekly streak). Focus on what, not how.
  4. /speckit.plan — Choose a language (e.g., Python or Node), a storage approach (a local JSON file), and a simple command structure.
  5. /speckit.tasks — Generate the task list and review the ordering.
  6. /speckit.implement — Let the agent build it, then run the app and tests.

Acceptance Criteria to Verify

  • Adding a habit persists it across runs.
  • Marking a habit complete twice in one day is idempotent.
  • The weekly summary shows the correct current streak per habit.
  • All logic is covered by passing unit tests.
Reflection

When a criterion fails, resist hand-editing the code. Update the spec or plan and regenerate. Notice how much faster iteration becomes when the spec — not the code — is your source of truth.

What's Next?

  • Add a new requirement to the spec (e.g., reminders) and watch how a single /speckit.plan → /speckit.tasks → /speckit.implement pass extends the app.
  • Combine SDD with the Agentic Workflows path to automate spec-driven changes in CI.
  • Bring Spec Kit into a real repository and trial it on your next feature.
